What it is

Utarus is the generic distillation of a production agent codebase — same skeleton, every domain concept stripped. What's left is a clean template you can fork into any vertical: support bots, internal-tools assistants, ops dashboards, research helpers.

The framework is opinionated about structure and silent on domain. You get:

Per-user agent pool

One pi-agent-core Agent per user, 24h TTL eviction, 100-agent cap. CLI and Telegram share the same pool.

YAML state files

One file per user at data/users/<slug>.yaml. The disk is the source of truth — re-read before any mutation.

Invite + admin onboard

INV- codes for new-user onboarding (Q&A flow), ADM- codes for granting admin rights. Single-use, revocable, audit-logged.

Skill framework

Markdown knowledge docs loaded on demand via use_skill. Each skill stays in context for the rest of the conversation.

TypeBox tools

Every tool parameter is validated by JSON-schema before your code runs. Hard rules live in the tool, not the prompt — the LLM cannot bypass them.

Fail-fast config

No fallback defaults. Missing env vars exit at startup with a clear message. State-file accessors throw on shape violations.

What it isn't: a vector store, a RAG framework, a multi-agent orchestrator, or a LangChain competitor. It does ship a first-class WebUI multi-chat + BinDrive. It's a small skeleton for stateful, tool-using agents with onboarding and browser access.

Quickstart

1. Clone and install

git clone <your-fork-url>/utarus.git
cd utarus
npm install
cp .env.example .env

2. Configure .env

# Required — the agent will not start without these
DEEPSEEK_API_KEY=sk-...
UTARUS_AGENT_NAME=Acme Support Bot
UTARUS_AGENT_PURPOSE=You are the support bot for Acme Corp. Help users file tickets, check order status, and answer FAQ. Decline anything off-scope.

# Required for Telegram
TELEGRAM_BOT_TOKEN=
TELEGRAM_ADMIN_IDS=123456789

# Optional — defaults to ./data
UTARUS_DATA_ROOT=./data

3. Run

npm run dev

Output:

Acme Support Bot starting...
Initializing DeepSeek model...
DeepSeek model: deepseek-v4-pro
TELEGRAM_BOT_TOKEN not set — running in CLI-only mode.
Acme Support Bot running. Type /help for commands.

acme-support-bot>

4. Try the flow

At the prompt, ask the agent to create a user:

acme-support-bot> set up a new user "Alice Wong", email alice@example.com

The agent calls init_user, creates data/users/alice-wong.yaml, and links your CLI session to that user.

To exercise the invite flow (in Telegram):

  1. Admin runs /invite for alice → gets INV-A1B2C3D4.
  2. Alice messages the bot with INV-A1B2C3D4.
  3. Bot asks Alice for her display name and email.
  4. Bot calls redeem_invite_code with Alice's Telegram ID + collected fields.
  5. Alice's state file is created, and her Telegram account is linked.

Architecture

Five layers, top to bottom. Each layer has one job and is independently replaceable.

┌─────────────────────────────────────────────────────────┐ │ Interface — CLI (readline) + Telegram (telegraf) │ Per-user agent key: cli_session | tg_<userId> └─────────────────────────────────────────────────────────┘ │ ┌─────────────────────────────────────────────────────────┐ │ Agent — src/agent.ts │ One pi-agent-core Agent per user (24h TTL, cap 100) │ System prompt composed from UTARUS_AGENT_NAME/PURPOSE └─────────────────────────────────────────────────────────┘ │ ┌─────────────────────────────────────────────────────────┐ │ Skills — src/skills/ │ Markdown knowledge docs loaded on demand via use_skill └─────────────────────────────────────────────────────────┘ │ ┌─────────────────────────────────────────────────────────┐ │ Tools — src/tools/ │ TypeBox-schematized. Hard rules live here as code. │ The LLM cannot bypass what the tool refuses. └─────────────────────────────────────────────────────────┘ │ ┌─────────────────────────────────────────────────────────┐ │ Persistence — src/state/ │ YAML per user at data/users/<slug>.yaml │ + invites.yaml, admin_codes.yaml, admin_ids.yaml └─────────────────────────────────────────────────────────┘

Why this layering

The framework's job is to make rules live in code, not prompts. A tool that refuses an action with a typed error is a tool the LLM cannot argue with. A system-prompt instruction is a suggestion the LLM can drift from. Every hard rule belongs in a tool.

The flip side is the skill layer: knowledge that helps the agent decide belongs in markdown docs loaded on demand. The agent loads the right skill before it touches the relevant tool — so the skill shapes the call, and the tool enforces the call.

Per-user agent key

Every request is routed to a per-user agent via a key:

  • cli_session — single shared CLI agent (one operator, one context)
  • tg_<userId> — one agent per Telegram user

Two humans chatting with the same Telegram bot get isolated conversation contexts. The framework never crosses the streams.

Data model

The framework reserves the load-bearing fields. Everything else is yours.

Writing a new tool

// src/tools/my-tool.ts
import { Type } from 'typebox';
import type { AgentTool } from '@earendil-works/pi-agent-core';

export function createMyTool(): AgentTool {
  return {
    name: 'my_tool',
    label: 'My Tool',
    description: 'One paragraph describing when the agent should call this.',
    parameters: Type.Object({
      foo: Type.String({ description: 'What foo means.' }),
    }),
    async execute(_id, raw) {
      const { foo } = raw as { foo: string };
      // Hard rules go here — the LLM cannot bypass what this code refuses.
      if (!foo.trim()) {
        return { content: [{ type: 'text', text: '❌ foo must be non-empty' }], details: null };
      }
      return {
        content: [{ type: 'text', text: `Did the thing with ${foo}.` }],
        details: { foo },
      };
    },
  };
}

Register it in src/agent.ts:

function frameworkTools(): AgentTool[] {
  const skillTool = createSkillTool();
  const userTools = createUserStateTools();
  const inviteTools = createInviteTools();
  const myTool = createMyTool();   // ← add
  return [skillTool, ...userTools, ...inviteTools, myTool];
}
Tool = hard rule. If a behaviour must always hold (KYC verified before activation, payment confirmed before upgrade, etc.), put it in a tool. The LLM cannot bypass typed validation.

Skills

Skills are markdown knowledge docs the agent loads on demand. The framework ships two:

  • getting-started — session protocol, state model, hard rules. Load at the start of any session that touches a user.
  • admin — invite and admin-onboard flows, code patterns, failure modes.

Adding a skill

  1. Write src/skills/knowledge/<id>.md.
  2. Register it in src/skills/registry.ts:
export const SKILLS: readonly Skill[] = [
  // ...
  {
    id: 'my-skill',
    name: 'My Skill',
    description: 'Load when ... <be specific — the LLM picks the skill from this description alone>',
    kind: 'knowledge',
    keywords: ['my', 'skill', 'keywords'],
  },
];

The description is the load-bearing field — the LLM never sees the file content until it calls use_skill, so the description has to be specific enough to pick the right skill from the catalog alone.

Extending the framework

Recipe 1: Phase-machine domain (e.g. onboarding lifecycle)

Add a top-level phase field under user or as a separate top-level key, then write an advance_phase tool that enforces gates in code:

// src/tools/phase.ts
export function createPhaseTools(): AgentTool[] {
  const advance: AgentTool = {
    name: 'advance_phase',
    label: 'Advance Phase',
    description: 'Move to the next phase. Enforces gate rules.',
    parameters: Type.Object({
      slug: Type.String(),
      to: Type.String({ description: 'Target phase.' }),
    }),
    async execute(_id, raw) {
      const p = raw as { slug: string; to: string };
      // load state, check gate, throw with specific error if violated
      // save state with new phase + log entry
    },
  };
  return [advance];
}

Because the gate runs in TypeScript — not in the system prompt — the LLM cannot negotiate it down. The error message tells the user (and the agent) exactly what state to fix.

Recipe 2: External integration (e.g. scrape / API call)

Wrap the external service in a tool. Read API keys from process.env; throw if missing.

function getApiKey(): string {
  const key = process.env.MY_API_KEY;
  if (!key) throw new Error('MY_API_KEY not set in environment');
  return key;
}

Recipe 3: Multiple agents in one process

Re-implement getOrCreateAgent in src/agent.ts with a different toolset per role. The framework's userKey scheme (cli_session / tg_<id>) is what isolates contexts — pick a different key scheme and you have separate agents.

Recipe 4: Database instead of YAML

Replace src/state/state-file.ts with a SQL/DocDB-backed implementation. The interface is the contract: loadState, saveState, listUserSlugs, stateExists, blankState, plus the invite/admin functions. Keep the same signatures and the rest of the framework doesn't notice.

Operations

Running headless

Set TELEGRAM_ONLY=true in .env to skip the CLI and run as a pure Telegram bot. Useful for pm2 / systemd deployment:

TELEGRAM_ONLY=true npm start

Granting admin to a new Telegram user

  1. Bootstrap admin (you) messages the bot with no special command.
  2. Run /admincode for ops team lead → get ADM-A1B2C3D4.
  3. Send the code out-of-band to the new admin.
  4. They send ADM-A1B2C3D4 to the bot.
  5. Bot validates, appends their Telegram ID to data/admin_ids.yaml, marks the code used.

Revoking an unused admin code

Run /revoke ADM-A1B2C3D4. Used codes cannot be revoked — the grant has already happened. To revoke an already-granted admin, remove their ID from data/admin_ids.yaml.
